# Diagnosis — Task 4.1 — Curve Label / Binding Inversion

> Generated by reading `Jhonny/data/CommonEvents.json` and
> `Jhonny/js/rmmz_objects.js` on 2026-06-21.

## Confirmed hypothesis: **H6 — Event-binding inversion (mouse + keyboard)**

None of the five pre-listed hypotheses (H1 coord swap, H2 file swap,
H3 condition inversion, H4 opcode drift, H5 TextPicture bake) explain the
bug. The bug is a sixth case: the in-game **Right arrow** (picture
`race/btn_direita` at x=624) is wired to the **Safe** common event (CE 11
`EV_OnSafe`), and the **Left arrow** (picture `race/btn_esquerda` at
x=336) is wired to the **Risk** common event (CE 12 `EV_OnRisk`). The
keyboard handler mirrors the same swap.

Per the **corrected spec convention** (`Corrida - Core Loop.md` §5 and
the project memory `curva-convention-inversion.md`), **Direita = Risk**
and **Esquerda = Safe**. Both bindings are inverted today.

## Evidence

### Placement (H1 rejected)

Show Picture dumps from CE 9 (`EV_RenderCurva`):

```
CE9[10] code=231 params=[43, 'race/btn_direita',   0, 0, 624, 328, 100, 100, 255, 0]
CE9[11] code=231 params=[44, 'race/btn_esquerda',  0, 0, 336, 336, 100, 100, 255, 0]
```

On a 816×624 screen, x=624 is the right half and x=336 is the left half.
**Coordinates match file semantics** (direita→right, esquerda→left), so
H1 is rejected.

### Picture files (H2 rejected)

```
$ ls Jhonny/img/pictures/race/ | grep -E 'btn_(direita|esquerda)'
race/btn_direita
race/btn_esquerda
```

Both files exist with names that match their semantic role. The asset
files themselves are correct (visual inspection of the screenshots in
the task spec confirms the arrows point the right way). H2 rejected.

### Mouse binding — THE BUG (CE 9 cmd 12, cmd 13)

```
CE9[12] code=355 params=['const p3 = $gameScreen.picture(43); if (p3) p3.mzkp_commonEventId = 11;']
CE9[13] code=355 params=['const p4 = $gameScreen.picture(44); if (p4) p4.mzkp_commonEventId = 12;']
```

Picture 43 (direita / right) → CE 11 (Safe). Picture 44 (esquerda /
left) → CE 12 (Risk). **Both inverted.**

Confirmed via `Jhonny/js/plugins/ButtonPicture.js`:
```
ButtonPicture.js:79  picture.mzkp_commonEventId = commonEventId;
ButtonPicture.js:89  $gameTemp.reserveCommonEvent(this.picture().mzkp_commonEventId);
```

So clicking pic 43 calls `reserveCommonEvent(11)` → EV_OnSafe runs.

### Keyboard binding — THE SAME BUG (CE 13 cmd 4)

`CE 13 EV_KeyInput` runs each frame and reads the active scene type:

```js
if ($gameVariables.value(102) === 0) {          // Sinal
  if (Input.isTriggered('down')) $gameTemp.reserveCommonEvent(11);  // Parar = Safe
  if (Input.isTriggered('up'))   $gameTemp.reserveCommonEvent(12);  // Furar = Risk
} else {                                         // Curva (VAR_SCENE_TYPE != 0)
  if (Input.isTriggered('right')) $gameTemp.reserveCommonEvent(11); // BUG: Right → Safe
  if (Input.isTriggered('left'))  $gameTemp.reserveCommonEvent(12); // BUG: Left → Risk
}
```

The **Sinal** branch is spec-correct (Up=Furar=Risk, Down=Parar=Safe —
mirrors the Sinaleiro convention). The **Curva** branch is inverted:
`right` should reserve CE 12 (Risk), `left` should reserve CE 11 (Safe).

### Opcode mapping (H4 rejected)

```
$ rg "Game_Interpreter.prototype.command(231|355|117|655)\b" rmmz_objects.js
Game_Interpreter.prototype.command117 = ...
Game_Interpreter.prototype.command231 = ...
```

231 = Show Picture ✓, 355 = Script ✓, 117 = Call Common Event ✓. No MV-era drift.

### TextPicture (H5 rejected)

`CE9[7]` does set TextPicture before `CE9[9]` Show Picture (id=61, the
"P_cena%" HUD). That HUD is **not** the label bug — its variable
(`\V[103]`, P_CENA) is written by CE 7 cmd 23 before CE 8/9 run, so it
bakes the correct value (this is the fix landed in Phase 3 bonus).
Labels for the buttons themselves are baked into the `race/btn_direita`
and `race/btn_esquerda` PNG files, not via TextPicture. H5 rejected.

### Condition inversion (H3 rejected)

There is no conditional branch on scene type gating the binding itself.
The `if/else` in `CE13[4]` switches between Sinal/Curva input **modes**,
but the CE ID values inside the Curva branch are the bug — not the
branch operator. H3 rejected.

## Required changes

Two patches — both swap CE IDs 11 ↔ 12 in the Curva branch only.

### Patch K — Mouse binding (CE 9 cmd 12, cmd 13)

Swap the CE ID assigned to each picture so direita→Risk, esquerda→Safe:

| Cmd | Before                                         | After                                          |
| --- | ---------------------------------------------- | ---------------------------------------------- |
| 12  | `picture(43).mzkp_commonEventId = 11;`        | `picture(43).mzkp_commonEventId = 12;`        |
| 13  | `picture(44).mzkp_commonEventId = 12;`        | `picture(44).mzkp_commonEventId = 11;`        |

### Patch L — Keyboard binding (CE 13 cmd 4)

Inside the `else` (Curva) branch of the existing inline JS, swap which
key reserves which CE:

```js
} else {
  if (Input.isTriggered('right')) $gameTemp.reserveCommonEvent(12);  // was 11
  if (Input.isTriggered('left'))  $gameTemp.reserveCommonEvent(11);  // was 12
}
```

The Sinal branch (`down`→11, `up`→12) stays unchanged.

## Implementation choice

Per task 4.2's "Implementation choice" matrix, H6 is closest to H1
(small, surgical JSON edit on specific params) but the params are
strings inside `code=355` Script commands rather than numbers in
`code=231` Show Picture. The same approach applies: a single Python
script that rewrites both CEs and is idempotent via pattern detection.

- File: `interaction/fase4/fix_curve_labels.py`
- Functions: `patch_k_mouse_bindings(ces)`, `patch_l_keyboard_bindings(ces)`
- Idempotency: each patch detects whether the corrected CE ID is already
  in place; if so, prints `skip` and leaves the JSON untouched.
- On second run both patches print `skip` and `git diff` is empty.

The patches must touch the param string of `code=355` commands
(verified: `Game_Interpreter.prototype.command355 = function() { ...
eval(this._params[0]); }` — params[0] is the JS source).
